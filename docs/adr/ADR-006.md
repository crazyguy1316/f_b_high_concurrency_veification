# ADR-006: 以 Redis 為交易真實來源之最終一致性模型與對帳補償策略

**狀態：** 已接受 (Accepted)  
**日期：** 2026-06-03  
**關係：** 正式化 ADR-001 §4 所隱含的一致性策略；細化 ADR-003 §2.3 所聲明的分散式事務處理邊界

---

## 1. 背景 (Context)

本專案架構為：

```
Nginx → FastAPI → Redis List Queue → Worker → Redis Lua → MySQL
```

Worker 的執行路徑存在不可消除的跨系統寫入序列：

```
① Redis Lua 扣減（原子，Redis 邊界內）  ✅
② MySQL INSERT（非同步，獨立系統）       可能失敗
```

由 ADR-003 §2.3 確認，步驟 ① 與 ② 之間不在任何原子性協議的保護範圍之內，屬於分散式事務問題。

面對此問題，系統有以下三條技術路線可選：

| 路線 | 機制 | 代價 |
| :--- | :--- | :--- |
| **強一致性** | Two-Phase Commit / XA Transaction | 跨系統鎖定，吞吐量崩潰；Redis 不原生支援 XA |
| **補償事務** | Saga Pattern（正向補償 + 反向回滾） | 複雜的補償邏輯；雙向失敗場景難以窮舉 |
| **最終一致性** | 接受暫時不一致，以對帳協議補完差異 | 存在對帳延遲窗口；需明確定義 Source of Truth |

本 ADR 記錄本專案選擇第三條路線的理由、邊界定義、補償機制規格，以及明確接受的技術債與風險。

### 1.1 驗證邊界（Validation Boundary）

本專案驗證目標為高併發交易流程之流量削峰、庫存扣減與最終一致性模型。

**本專案不驗證：**

*   完整訂單生命週期管理
*   分散式強一致性協議
*   多資料中心災難復原
*   金融級帳務追蹤
*   Event Sourcing / CDC 架構

因此 Reconciliation 僅需滿足本專案定義之資料正確性標準（成交數量一致性為主要驗證目標），不追求生產級別的審計能力。

---

## 2. 決策 (Decision)

### 2.1 放棄分散式強一致性協議

**本專案明確放棄以下機制：**

*   **Two-Phase Commit / XA Transaction**：Redis 不原生支援 XA；引入 XA Coordinator 將使架構複雜度倍增，且跨系統鎖定期間的吞吐量將從萬級 RPS 降至百級，與本專案核心驗證目標相悖。
*   **Saga Pattern（補償事務）**：在「Lua 扣減成功 → MySQL 失敗 → 補償回滾 Redis 扣減」的路徑中，補償操作本身亦可能失敗，引發更複雜的雙向不一致場景，且補償邏輯維護成本超出本驗證專案的合理範疇。

**決策依據**：本專案的核心目標是驗證高吞吐量架構，而非生產環境的強一致性保障。以效能換取一致性強度，在本驗證情境下是明確且合理的取捨。

### 2.2 確立 Redis 為唯一交易真實來源（Source of Truth）

**定案：** `ticket:success:orders` Redis Hash 是本專案驗證範圍內的交易成功事實來源

**理由：**

1. **Lua 原子寫入確保可信度**：成交記錄由 Redis Lua Script 在單線程排他執行中完成，不存在部分寫入或競態條件。Lua 成功即代表庫存已正確扣減且防重已記錄，結果具備最高可信度。
2. **MySQL 的角色為衍生資料**：MySQL `orders` 表的唯一職責是將 Redis 中的成交事實轉化為關係型持久化記錄，其內容應當且只應當是 Redis Source of Truth 的子集（在 Reconciliation 完成後應等於全集）。Reconciliation 僅驗證成交數量一致性。成交會員集合比對僅作為補償流程中的輔助資料來源，不作為獨立驗證指標。
3. **架構現實**：Redis 在整個交易流程中是唯一跨越「交易決策」邊界的系統；MySQL 的寫入發生在決策之後。以決策點的系統為真實來源，是符合架構語義的選擇。

### 2.3 Redis 與 MySQL 的責任邊界定義

| 責任項目 | 系統 | 說明 |
| :--- | :--- | :--- |
| **交易決策**（是否成交）| Redis Lua | 原子執行，不可更改，不得回滾 |
| **成交記錄**（誰成交了）| Redis `ticket:success:orders` Hash | Source of Truth，Reconciliation 的比對基準 |
| **持久化憑證**（可查詢的訂單）| MySQL `orders` 表 | 衍生資料，允許暫時落後於 Redis |
| **差異補償**（缺失訂單補寫）| `reconcile.py` 對帳腳本 | 以 Redis Hash 重建 MySQL 缺失記錄 |

**邊界禁區：**

*   MySQL INSERT 失敗時，**嚴禁反向撤銷 Redis Lua 的扣減結果**。Redis Source of Truth 的地位一旦被反向修改，整個一致性模型即告崩潰。
*   MySQL 的寫入失敗**不觸發交易回滾**，視為持久化延遲，由對帳補償。

### 2.4 最終一致性的定義與時序約束

本專案對「最終一致性」的操作性定義如下：

> **在 `ticket:request:queue` 完全清空（Worker 已消費完畢）且 `reconcile.py` 執行完成後，MySQL `orders` 表的成功訂單筆數必須等於 Redis `ticket:success:orders` Hash 的欄位數量。**

此定義隱含以下時序約束：

1. 對帳腳本 **必須且只能** 在佇列清空後執行，禁止在 Worker 尚未處理完畢時比對（比對的是終態，而非中間態）。
2. 對帳腳本必須等待足夠時間讓 Worker 完成所有 MySQL 落庫重試後再執行比對。

### 2.5 Reconciliation 機制規格

**觸發條件（必要，非可選）：**

*   每次整合壓測（`run_sandbox_test.ps1`）的壓測階段結束後，**強制執行**一次 `reconcile.py`。
*   `reconcile.py` 執行失敗（`AssertionError` 或連線異常）視為測試流程失敗，不可略過。

**執行邏輯：**

```
Step 1: 等待 ticket:request:queue 長度為 0
         （確認 Worker 已消費完全部佇列請求）
Step 2: 查詢 Redis HLEN ticket:success:orders  → redis_count
Step 3: 查詢 MySQL SELECT COUNT(*) FROM orders WHERE status = 'SUCCESS'  → mysql_count
Step 4: 比對 redis_count == mysql_count
         若相等  → [RECONCILE_SUCCESS]，流程通過
         若不等  → 進入補償邏輯（見下）
Step 5（補償）: 從 Redis Hash 讀取全部成交 member_id
               與 MySQL 現有記錄比對，找出缺失的 member_id
               以 INSERT IGNORE 補寫缺失訂單至 MySQL
Step 6（補償後驗證）: 重新執行 Step 3 與 Step 4
               若補償後仍不等 → [RECONCILE_FAILURE]，視為系統故障
```

**責任歸屬：**
`INTEGRATION_TESTER` 負責執行此腳本，並將完整輸出寫入 `log/sandbox_runtime.log`，供 `LOG_ANALYZER_RECOVERY` 進行後續分析。

### 2.6 可接受的不一致 vs 系統故障的邊界定義

| 場景 | 分類 | 處理 |
| :--- | :--- | :--- |
| 壓測結束後，MySQL 筆數 < Redis Hash 筆數 | **可接受的暫時不一致** | 由 `reconcile.py` 補償後視為通過 |
| 補償後 MySQL 筆數 = Redis Hash 筆數 | **一致性已達成** | `[RECONCILE_SUCCESS]` |
| MySQL 筆數 > Redis Hash 筆數 | **系統故障**（超賣或幽靈訂單）| `[RECONCILE_FAILURE]`，立即停止分析，上報 `LOG_ANALYZER_RECOVERY` |
| 補償後仍有差異（補寫失敗）| **系統故障** | `[RECONCILE_FAILURE]` |
| Redis `ticket:success:orders` 資料遺失 | **不可恢復的系統故障** | Source of Truth 已消失，無法對帳，須人工介入 |

---

## 3. 理由 (Rationale)

1. **吞吐量目標決定一致性模型選擇**：本專案核心驗證指標為萬人級別高併發吞吐量。Two-Phase Commit 的跨系統鎖定機制會在高併發下形成吞吐量瓶頸，從根本上違背驗證目標。最終一致性模型是在「驗證高吞吐架構」這一前提下，唯一符合目標的一致性策略。

2. **Lua 的原子性提供了可信的事實基礎**：Redis Lua Script 的單線程執行特性，確保了每一筆成交記錄都是在無競態條件下寫入的。以具備最高原子性保障的系統作為 Source of Truth，在架構語義上是自洽的。

3. **Reconciliation 比 Saga 更適合本驗證情境**：Saga Pattern 需要設計完整的補償邏輯樹，包括補償失敗的補償、超時的處理等，其複雜度遠超本驗證專案的合理範疇。`reconcile.py` 的直接比對 + 補寫邏輯，在本單機驗證環境中已足夠覆蓋所有失敗場景。

4. **MySQL 的聯合唯一索引提供最終防護**：`orders` 表建有 `UNIQUE KEY uk_member_event (member_id, event_id)` 聯合唯一索引。即使補償機制的 INSERT 操作因任何原因重複執行，資料庫層面的唯一約束將自動防止重複訂單產生。

---

## 4. 後果 (Consequences)

### 4.1 優點

*   系統在高併發下維持最大吞吐量，不因跨系統同步鎖定而降級。
*   Worker 的 MySQL 落庫失敗不影響交易決策的正確性，故障隔離清晰。
*   Reconciliation 機制提供了可驗證的最終一致性保障，測試結果具備確定性。

### 4.2 明確接受的技術債與風險

> [!CAUTION]
> 以下風險已被明確識別並有意識地接受。接受這些風險的前提是：**本專案為單機驗證環境，而非生產系統**。在任何生產化規劃中，必須重新評估以下每一項風險。

#### 風險一：Redis 為單一真實來源（Single Point of Truth）的風險

*   **風險描述**：Redis 是記憶體資料庫，若發生宿主機斷電、Docker 容器崩潰或 OOM Killer 介入，且未啟用 AOF/RDB 持久化，`ticket:success:orders` Hash 可能完全遺失。Source of Truth 一旦遺失，`reconcile.py` 無從比對，已成交記錄將永久不可恢復。
*   **接受理由**：本專案為單機 Docker 驗證環境，Redis 容器運行期間的資料存活率足以完成壓測與對帳流程。生產環境若採用此架構，**必須**啟用 Redis AOF 持久化或採用 Redis Cluster with Replica，此為已知的架構演進需求。

#### 風險二：Reconciliation 延遲期間的不一致暴露窗口

*   **風險描述**：從 Worker 完成 Lua 扣減到 `reconcile.py` 執行完成之前，存在一段時間窗口，在此期間 MySQL 的訂單數量少於 Redis 的成交數量。若有外部系統在此窗口內查詢 MySQL 訂單數，將得到不完整的結果。
*   **接受理由**：本專案為封閉的壓測驗證環境，不存在並發外部查詢。對帳窗口的不一致暴露風險在本情境下不構成實際影響。

#### 風險三：Worker Crash 導致 MySQL 持續落後的風險

*   **風險描述**：若 `background_worker` 容器在壓測中途崩潰，大量已完成 Lua 扣減的請求將因 Worker 不可用而永遠無法落庫，直至 Worker 重啟。重啟後若佇列已清空（`BRPOP` 已超時），這批請求的 MySQL 落庫將永遠不會發生，只能依賴 `reconcile.py` 補寫。
*   **接受理由**：`reconcile.py` 的補償機制設計上覆蓋了此場景（從 Redis Hash 遍歷重建所有缺失訂單），故 Worker Crash 不會導致永久資料遺失。但補償後的 MySQL 記錄將缺少原始的 `created_at` 時間戳，訂單時序資訊將不準確。

#### 風險四：MySQL INSERT 失敗不觸發重試的風險

*   **風險描述**：當前 Worker 設計在 MySQL INSERT 失敗時，僅記錄錯誤並繼續處理下一條佇列請求，不對失敗的 INSERT 進行重試。這意味著 Reconciliation 是此類失敗的唯一補救手段。
*   **接受理由**：本驗證環境下的 MySQL 連線失敗率極低（有防禦性重試初始化機制，見 ADR-005），`reconcile.py` 足以覆蓋殘餘失敗。生產環境中應加入 Dead Letter Queue 或 INSERT 重試機制以降低對 Reconciliation 的依賴。

#### 風險五：未採用強一致性協議所帶來的架構演進限制

*   **風險描述**：若本系統未來需要橫向擴展為真正的分散式系統（多 Redis 節點、多 Worker 實例、多 MySQL Shard），當前以單一 Redis Instance 為 Source of Truth 的模型將面臨以下挑戰：
    *   跨 Redis 節點的 Lua Script 無法保持原子性（Lua Key 必須在同一 Slot，見 ADR-002）。
    *   多個 Worker 實例並發寫入 MySQL 時，`reconcile.py` 的執行時機與鎖定策略需重新設計。
    *   Redis Cluster 模式下，`HLEN ticket:success:orders` 等全量查詢操作的語義需重新評估。
*   **接受理由**：本專案明確定位為單機 Docker 驗證環境。上述擴展場景不在本專案的驗證範疇之內。此為已知的架構演進天花板，在進行生產化規劃時必須作為第一優先項重新評估。

## 5. 未採用模式的正式封存

以下模式已在本 ADR 評估後明確排除，後續工單不得重新提案，除非專案目標發生根本性變更：

| 排除模式 | 排除理由 |
| :--- | :--- |
| Two-Phase Commit (2PC) | Redis 不支援 XA；跨系統鎖定與高吞吐目標不相容 |
| XA Transaction | 同上；引入 XA Coordinator 使架構複雜度不可接受地增加 |
| Saga Pattern（補償事務）| 補償邏輯樹複雜度超出本驗證專案的合理範疇；補償失敗的處理引入額外風險 |
| MySQL 作為 Source of Truth | MySQL 寫入發生在交易決策之後，以衍生資料作為真實來源在架構語義上不自洽 |
