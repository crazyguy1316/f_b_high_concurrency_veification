# ADR-003: Async I/O 連線庫選型與 Lua 腳本原子性防護邊界

**狀態：** 已接受 (Accepted)  
**日期：** 2026-05-19  
**關係：** 細化 ADR-001「原子性快取阻擊層」的 Python 應用層實作約束；明確 Lua 原子性邊界，並承接 ADR-001 之分散式一致性策略

---

## 1. 背景 (Context)

ADR-001 確立了以 Redis Lua Script 作為原子性快取阻擊層的架構方向，但對 Python 應用層所應採用的 I/O 驅動庫，以及協程層面（Asyncio）可能帶來的並發狀態交錯問題，尚未明確定義強制性約束。

在 FastAPI（基於 Python `asyncio` 的 ASGI 框架）中，若協程切換（Context Switch）發生在組合式狀態讀寫（Read-then-Write）之間，即使單獨的 `await` 操作各自是原子性的，整個組合操作在宏觀上仍然是非原子性的，依然存在競態條件（Race Condition）。

然而，必須清楚認識 **Lua 腳本原子性的作用邊界**：Lua 的原子性保護範圍僅限於 **Redis 內部**。當 Worker 在 Lua 成功返回 `SUCCESS` 後，繼而執行 MySQL 非同步落庫時，這兩個操作跨越了兩個獨立的狀態儲存系統，Lua 的原子性保護在此處完全失效。此時問題的性質已從 **Race Condition（競態條件）** 升級為 **Distributed Transaction（分散式交易）**：

```
Lua 扣減 → Redis: ticket:success:orders 已記錄成交  ✅
           ↓
       MySQL INSERT 失敗（網路抖動、容器崩潰等）  ❌
           ↓
  Redis 已成交 vs MySQL 沒有紀錄  → 資料不一致
```

本 ADR 需同時處理兩個層次的問題，並與 ADR-001 的一致性策略形成完整閉環。

---

## 2. 決策 (Decision)

### 2.1 連線庫選型定案

全面採用以下 Async I/O 驅動庫：

| 中間件 | 選型定案 | 淘汰替代方案 |
| :--- | :--- | :--- |
| **Redis** | `redis.asyncio` | `aioredis`（已合併入官方庫）|
| **MySQL** | `aiomysql` | `databases`、`SQLAlchemy async`（過重）|

**選型理由**：兩者均為非阻塞（Non-blocking）驅動，能夠在 FastAPI 的 asyncio 事件循環中最大化 RPS，避免因同步阻塞調用榨乾事件循環執行緒。

### 2.2 原子性防護邊界（Anti-Pattern Guardrail）

確立以下強制性編碼約束，作為不可逾越的防護邊界：

*   **禁止組合式狀態變更**：嚴禁在 Python 業務邏輯層面以 `await redis.get(key)` 加 `await redis.set(key)` 的組合方式執行任何涉及庫存、身份防重等狀態變更。此模式在 Asyncio 協程交錯切換時，兩個 `await` 之間的時間窗口將產生競態條件。

*   **強制走 Lua 腳本路徑**：所有涉及「庫存扣減」、「防重複購買（Idempotency Check）」等複合狀態變更，一律強制透過 `await redis.eval()` 或 `await redis.evalsha()` 調用預載入的 Lua 腳本。藉由 Redis 單線程核心的天然排他性，確保 Redis 內部複合狀態變更具備單次執行的原子性。

### 2.3 Lua 原子性邊界聲明與分散式事務策略（引用 ADR-001）

**邊界聲明**：Lua 的原子性保護範圍止於 Redis 服務端內部。`Lua SUCCESS → MySQL INSERT` 這條執行路徑跨越兩個獨立系統，不在 Lua 原子性的保護之內，屬於分散式事務（Distributed Transaction）問題。

**本專案採用之一致性策略**（定案於 ADR-001 §4，更新版本）：

> 放棄分散式強一致性（Two-Phase Commit / Saga）以換取效能，接受「最終一致性（Eventual Consistency）」模型，並以以下策略閉合缺口：

| 角色 | 系統 | 說明 |
| :--- | :--- | :--- |
| **交易真實來源（Source of Truth）** | Redis `ticket:success:orders` Hash | Lua 原子寫入，視為唯一可信的成交事實 |
| **衍生持久化資料（Derived Persistence）** | MySQL `orders` 表 | Worker 非同步落庫，允許暫時落後於 Redis |
| **補償機制（Compensation）** | 對帳腳本 `reconcile.py` | 若 MySQL 筆數 < Redis Hash 筆數，以 Redis Success Hash 重建 MySQL 缺失訂單 |

**Worker 落庫失敗的處理約束**：
*   MySQL INSERT 失敗時，**不得**回滾 Redis 的 Lua 扣減結果（否則將引發更複雜的雙向補償問題）。
*   MySQL INSERT 失敗視為「暫時落庫延遲」，由後續對帳協議補單，而非系統錯誤。
*   對帳腳本必須等待 `ticket:request:queue` 完全清空（Worker 處理完畢）後才執行，確保比對的是終態數據。

---

## 3. 理由 (Rationale)

1. **根除協程切換競態（Redis 邊界內）**：Redis 的 Lua 腳本在服務端以單線程方式執行，執行期間不處理任何其他命令。在本專案中是確保 `asyncio` 多協程並發環境中，確保多步驟 Redis 操作完全不被打斷的技術手段。此策略解決的是 **Redis 邊界內的 Race Condition**。

2. **接受最終一致性以換取效能（Redis→MySQL 邊界）**：引入 Two-Phase Commit 或 Saga Pattern 可實現跨系統強一致性，但代價是大幅降低吞吐量並引入複雜的補償邏輯。本專案驗證目標為高併發吞吐效能，故選擇接受最終一致性模型：以 Redis 為唯一成交真實來源，MySQL 為衍生持久化資料，以對帳協議補完差異。

3. **職責邊界清晰**：Python 應用層（FastAPI/Worker）僅負責「非狀態型」的讀取操作（如查詢訂單狀態）與網絡 I/O 調度；所有「狀態型」的讀改寫（Read-Modify-Write）邏輯，集中封裝於 Redis Lua 腳本中，形成明確的職責分層。

4. **效能最大化**：`redis.asyncio` 與 `aiomysql` 不阻塞 asyncio 事件循環，使 FastAPI 能在單個進程內高效處理大量並發連線，充分發揮 ASGI 框架的吞吐量上限。

---

## 4. 後果 (Consequences)

*   **優點**：
    *   從根源杜絕了超賣（Overselling）在 Redis 層面的可能性；Race Condition 問題在 Redis 邊界內被完全消除。
    *   應用層代碼更加簡潔：業務邏輯 Worker 只需解讀 Lua 腳本的回傳值（`SUCCESS` / `SOLD_OUT` / `DUPLICATE_ORDER`），無需實作複雜的鎖機制。
    *   最終一致性策略（Redis SoT + 對帳補單）相比分散式強一致性方案，在效能上具備壓倒性優勢，符合本專案的高吞吐驗證目標。

*   **挑戰與配套**：
    *   **[已知風險] Lua Success → MySQL Fail**：此場景在最終一致性模型下屬於「可接受的暫時不一致」，不視為超賣。但對帳腳本（`reconcile.py`）**必須**具備從 Redis `ticket:success:orders` Hash 重建 MySQL 缺失訂單的補償能力，否則此風險將成為永久資料遺失。
    *   **[硬性約束] 不得反向回滾 Redis**：MySQL INSERT 失敗時，嚴禁嘗試反向撤銷 Redis 的 Lua 扣減結果。此操作將破壞 Redis 的 Source of Truth 地位，引發更嚴重的雙向不一致問題。
    *   **Lua 腳本維護成本**：業務邏輯的核心修改需同步更新 Lua 腳本，且調試難度高於純 Python 代碼。需由 `ARCHITECT_DISPATCHER` 在工單中明確標注 Lua 腳本的修改點，並由 `REVIEWER_REFACTOR` 進行雙重確認。
    *   **[審查清單] 禁令執行**：需在 `REVIEWER_REFACTOR` 的靜態審查清單中，加入「是否有違規的組合式 `await redis.get/set` 調用」以及「MySQL 失敗時是否有錯誤嘗試回滾 Redis」兩項強制性審查項目。
