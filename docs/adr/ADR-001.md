# ADR-001: 包含 Client-to-Server 串接的高併發環境下技術選型與非同步解耦架構驗證

**狀態：** 已接受 (Accepted)
**關係：** 奠定系統邊緣防禦、核心快取阻擊與非同步落庫之基礎通訊模式

## 1. 背景 (Context)
系統需在單機 Docker 封閉測試環境下，模擬大規模瞬時高併發搶票場景。傳統架構若讓海量前端連線直接直擊後端 Application 與 RDBMS（MySQL），將引進以下物理限制：
1. 靜態資源（HTML/JS）讀取耗盡伺服器執行緒。
2. 複雜加密驗證（如 JWT）榨乾 CPU 算力。
3. 關係型資料庫因磁碟 I/O 瓶頸與鎖競爭（Lock Contention）導致連線池耗盡及系統崩潰。

此外，由於本地實驗室環境無公有雲 CDN 支援，必須在架構內部實作等價的流量清洗機制。

## 2. 決策 (Decision)
為實現多層漏斗防禦（Funnel Architecture）並最大化壓測吞吐量，採取以下技術選型與架構約束：

*   **網關邊緣防線（Edge Ingress Gateway）**：
    採用 **反向代理網關 (Nginx)** 作為單一入口。實作「靜態資源完全卸載」，由網關直接響應前端網頁；並於動態 API 端點配置「基於 IP 的速率限制（Rate Limiting）」，在邊界直接阻斷惡意刷票流量。
*   **應用層輕量驗證（Lightweight Authentication）**：
    前端與 App 間通訊全面精簡。會員 ID 採用 `int` 型態以優化記憶體索引；認證機制因不是驗證重點，為配合環境運算資源不足，故採用不具備生產環境安全性字串拼接型超輕量 Token，以最低運算成本過濾非法請求（回應 401），模擬無效流量不進入核心佇列。
*   **輕量消息緩衝層（Message Queue）**：
    為配合測試環境運算資源不足，選用 **Redis List** 作為輕量級消息緩衝器（Lightweight Queue Buffer），用於模擬 Message Queue 在削峰填谷架構中的角色。驗證通過的搶票請求透過 `LPUSH` 推入佇列 `ticket:request:queue`，利用其簡單高效的特點緩衝瞬間流量，再由後台背景 Worker 透過 `BRPOP` 進行非同步消費。
*   **原子性快取阻擊層（Cache Ingress）**：
    所有的庫存檢查、防重放（防重複購買）與預扣邏輯，必須封裝於 **Redis Lua Script** 中，由背景 Worker 提取佇列請求後進行單執行緒排他執行，從根源杜絕超賣。
*   **持久化終態資料庫（Persistence DB）**：
    採用 **MySQL 8.0** 作為最終交易紀錄憑證，由後台 Worker 在快取層扣減成功後，以穩定速率非同步寫入（INSERT），達成數據最終一致性。

## 3. 理由 (Rationale)
1. **職責分離與效能優化**：Nginx 處理靜態與限流、App 處理輕量驗證與快速入隊、Worker 與 Redis 處理原子扣庫存與落庫，各層各司其職，最大化單機環境下的 RPS 天花板。
2. **運算成本控制**：放棄重型 MQ（如 Kafka/RabbitMQ）與重型加密演算法，將多餘的 CPU 與記憶體資源 100% 留給高併發連線壓測。
3. **投資與架構保護**：此網關與佇列的職責劃分，未來在系統橫向擴展（Scale-up）上雲端時，可平滑移轉至實體 CDN、WAF 與分散式微服務，架構具備高度延續性。

## 4. 後果 (Consequences)
*   **優點**：
    *   系統具備極高的邊界防禦與崩潰保護能力，核心業務層（App/DB）獲得完全解耦。
    *   利用 k6 進行內網壓測時，可撇除外部網路延遲雜訊，精準測出後端架構的純粹效能極限。
*   **挑戰與配套**：
    *   **非同步落庫延遲**：k6 壓測結束時，MySQL 資料可能尚未完全寫入。測試驗證時，必須實作「終態資料對帳協議（Data Reconciliation）」，等待 Redis List 佇列完全清空且 Worker 處理完畢後，比對 MySQL 實體總筆數與 Redis 中獎雜湊表（`ticket:success:orders`）記錄，兩者必須絕對等於扣減庫存量。資料正確性則以 Redis 為交易真實來源（Source of Truth），MySQL 為衍生持久化資料；若發現差異，以 Redis Success Hash 重建 MySQL。
