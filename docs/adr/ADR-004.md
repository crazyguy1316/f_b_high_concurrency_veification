# ADR-004: Waterfall Flow 協作模式與 INTEGRATION_TESTER 專職 Agent 引入

**狀態：** 已接受 (Accepted)  
**日期：** 2026-05-19  
**關係：** 擴展 Multi-Agent 協作管道，細化測試驗證閉環結構

---

## 1. 背景 (Context)

在 Sprint 1 的初始 Multi-Agent 協作管道中，`REVIEWER_REFACTOR` 身兼代碼靜態走查、單元測試審查，以及隱式地宣稱「壓測驗證通過」等多重職責。此設計存在以下問題：

1. **職責膨脹（Responsibility Bloat）**：代碼的「靜態品質」（語法正確性、OOP 原則）與「實體環境併發表現」（Docker 容器壓測、資料一致性）是截然不同的品質維度，不應由同一個 Agent 混合處理。

2. **除錯範圍過大（High Blast Radius）**：若沒有獨立的整合測試階段，在壓測出現問題時，無法區分故障根因是「代碼邏輯錯誤」還是「運行期環境競態條件」，除錯成本極高。

3. **缺乏自動化閉環能力**：壓測失敗時，無法自動產生標準化的日誌輸入，驅動 `LOG_ANALYZER_RECOVERY` 進行精準的 Healing 反饋。

---

## 2. 決策 (Decision)

### 2.1 確立 Waterfall Flow 工作原則

採用嚴格的瀑布式流程（Waterfall Flow）作為 Multi-Agent 協作的強制工作模式：**每個 Agent 在完成本階段職責並交付產出物後，即刻停止，不得跨足下一階段的工作**。

**完整工作管道（Pipeline）定義：**

```
ARCHITECT_DISPATCHER
  → (工單 JSON)
GENERAL_OOP_IMPLEMENTER
  → (src_staging/ 代碼)
REVIEWER_REFACTOR
  → (靜態審查通過，同步至 src/)
INTEGRATION_TESTER
  → (log/sandbox_runtime.log 日誌)
LOG_ANALYZER_RECOVERY
  → (Healing JSON / [PASS] 認證)
```

### 2.2 新增 INTEGRATION_TESTER 專職 Agent

新建 `INTEGRATION_TESTER` Agent（`agents/integration_tester.json` v1.0.0），職責定義如下：

| 項目 | 規格 |
| :--- | :--- |
| **唯一輸入源** | `src/`（由 `REVIEWER_REFACTOR` 放行後的穩定代碼）|
| **執行環境** | 實體 Docker Compose 沙盒 |
| **核心職責** | 容器編排啟動、k6 高併發壓力測試、資料一致性對帳審計 |
| **唯一輸出目標** | `log/sandbox_runtime.log`（全量 Stdout + Stderr）|
| **禁止事項** | 不修改任何 `src/` 代碼；不對測試結果自行評估，一律輸出完整日誌由下游 Agent 分析 |

### 2.3 REVIEWER_REFACTOR 職責收斂

`REVIEWER_REFACTOR` 職責收斂至：
- **代碼靜態走查**（語法、OOP 原則、Anti-Pattern 偵測）
- **Mock 單元測試審查**（不需要啟動實體容器）
- **放行同步**（審查通過後，將 `src_staging/` 代碼同步至 `src/`）

**不再**負責任何 Docker 沙盒操作或壓測相關事務。

---

## 3. 理由 (Rationale)

1. **縮減除錯範圍（Blast Radius Reduction）**：若代碼在 `REVIEWER_REFACTOR` 階段被阻斷（語法/邏輯不合格），則問題根因可精確定位為「代碼品質問題」。若代碼通過靜態審查後在 `INTEGRATION_TESTER` 的壓測中失敗，則問題範圍可收斂至運行期行為與環境整合層。兩個維度嚴格分層，大幅降低排查成本。

2. **促成無人值守 Healing 閉環**：`INTEGRATION_TESTER` 產出的標準化 `sandbox_runtime.log` 是 `LOG_ANALYZER_RECOVERY` 的唯一合法輸入源。此設計使整個驗證流程具備全自動的「錯誤偵測 → 根因分析 → 修復指令生成 → 回退至實作員重構」的閉環能力。

3. **單一職責原則（SRP）在 Agent 層面的踐行**：將「我知道代碼是否正確」與「我知道系統在壓力下是否穩定」這兩個截然不同的關注點，分配給兩個獨立的 Agent，提升系統的可維護性與可演進性。

---

## 4. 後果 (Consequences)

*   **優點**：
    *   Multi-Agent 管道具備嚴格的品質閘門（Quality Gate），確保每個階段的交付物符合品質標準後才進入下一階段。
    *   `LOG_ANALYZER_RECOVERY` 的輸入源得到標準化，分析精準度大幅提升。
    *   整個驗證流程可在無人值守的情況下自動執行，並在失敗時自動生成修復指導。

*   **挑戰與配套**：
    *   **管道延遲增加**：引入獨立整合測試階段後，完整走一遍 Pipeline 的時間成本有所增加。但考慮到換取的除錯精準度提升，此成本可接受。
    *   **Agent 協作協定維護**：`ARCHITECT_DISPATCHER` 需在工單中明確標注每個 Agent 的職責邊界，避免職責蔓延。`SD_SI_PROJECT_ASSISTANT` 負責定期進行 Agent 交叉審查（Cross-Audit），確保 IO Contract 不產生斷層。
