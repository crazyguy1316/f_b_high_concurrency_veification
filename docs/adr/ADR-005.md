# ADR-005: Docker 容器啟動競態條件防禦模式（Sequenced Bootstrapping）

**狀態：** 已接受 (Accepted)  
**日期：** 2026-05-19  
**關係：** 細化 Sprint 2 Docker Compose 沙盒的實體啟動協議，基於整合測試實際崩潰診斷

---

## 1. 背景 (Context)

在 Sprint 2 `INTEGRATION_TESTER` 執行沙盒整合測試期間，發現 Docker Compose 的標準同步啟動策略（`docker-compose up --build`）在微服務依賴鏈上存在兩個關鍵缺陷，導致測試環境無法穩定運行：

### 問題一：Python 相對導入失效（`ImportError`）

**錯誤現象：**
```text
ImportError: attempted relative import with no known parent package
```

**根因**：`Dockerfile.worker` 以 `python src/app/worker.py` 直接啟動，Python 以「主程式模式（`__main__`）」執行，導致相對導入（Relative Import）語法（`from .services import ...`）在缺乏包上下文的情況下失效。

### 問題二：MySQL 連線競態條件（`Connection Refused`）

**錯誤現象：**
```text
pymysql.err.OperationalError: (2003, "Can't connect to MySQL server on 'db_mysql'")
```

**根因**：Docker Healthcheck 成功並不等同於應用程式已完全 Ready。`app_server` 與 `background_worker` 容器在此窗口內搶先建立連線，導致 Socket 連線被拒絕。此外，`main.py` 缺少 `import asyncio` 語句，導致首次重試時觸發 `NameError` 二次崩潰。

---

## 2. 決策 (Decision)

### 2.1 Dockerfile 啟動命令修正

**修正前（錯誤模式）：**
```dockerfile
CMD ["python", "src/app/worker.py"]
```

**修正後（定案模式）：**
```dockerfile
CMD ["python", "-m", "src.app.worker"]
```

**約束**：所有 Python 模組化應用的 Dockerfile 啟動命令，**必須**以 `-m` 旗標以包（Package）方式調用，以維持相對導入的包上下文完整性。

### 2.2 應用層防禦性重試機制（Defensive Retry）

在 `main.py` 與 `worker.py` 的 MySQL 連線池初始化處，**必須**實作以下防禦性重試機制：

```python
import asyncio  # 必須顯式導入

async def create_db_pool_with_retry(max_retries=5, delay_seconds=2):
    for attempt in range(max_retries):
        try:
            pool = await aiomysql.create_pool(...)
            return pool
        except Exception as e:
            if attempt < max_retries - 1:
                await asyncio.sleep(delay_seconds)
            else:
                raise
```

**規格約束**：
- 最大重試次數：**5 次**
- 重試間隔：**2 秒**
- 必須顯式包含 `import asyncio`（禁止依賴隱式導入）

### 2.3 沙盒自動化腳本分步引導啟動（Sequenced Bootstrapping）

`run_sandbox_test.ps1` 採用分步啟動協議，取代原先的一次性 `docker-compose up --build`：

```
步驟 1: docker-compose up -d db_mysql redis
         → 僅啟動基礎設施層（資料庫與快取）

步驟 2: 輪詢等待
         → 循環執行 docker inspect，
           直到 db_mysql 的 Health Status 變為 "healthy"

步驟 3: docker-compose up --build -d app_server background_worker nginx
         → 確認底層依賴完全就緒後，再編譯並啟動應用層與網關
```

**理由**：此模式確保基礎設施層（MySQL、Redis）在應用層容器啟動前，已完成 TCP 埠綁定並進入完全服務就緒狀態，從根本消除連線競態條件窗口。

---

## 3. 理由 (Rationale)

1. **實體驗證驅動（Evidence-Based）**：此 ADR 中的所有決策，均基於 Sprint 2 整合測試中真實觀測到的崩潰日誌，而非預防性假設。每一條約束都對應一個具體的生產環境失敗模式。

2. **分層啟動降低不確定性**：Docker Compose 的 `depends_on: condition: service_healthy` 機制在某些環境版本下行為不一致。採用腳本顯式輪詢 `docker inspect` 狀態，是在 Windows/WSL2 環境下最可靠的容器就緒判斷手段。

3. **標準化可複用**：此防禦性重試模式與分步啟動協議，一旦定案，應作為所有後續 Sprint 的沙盒測試標準程序，不得因「在我的機器上能跑」而省略。

---

## 4. 後果 (Consequences)

*   **優點**：
    *   整合測試環境啟動穩定性大幅提升，消除了環境不穩定帶來的偽陽性（False Negative）失敗。
    *   連線競態條件的防禦模式被標準化為可複用的模板，未來新增服務時可直接套用。

*   **挑戰與配套**：
    *   **啟動時間增加**：分步啟動協議比一次性啟動多出 10~30 秒等待時間（視 MySQL 初始化速度而定）。這是換取環境穩定性的可接受成本。
    *   **腳本維護**：若未來調整容器依賴關係，必須同步更新 `run_sandbox_test.ps1` 的分步啟動序列，並在 `si_env_changes.md` 中追加異動紀錄。
