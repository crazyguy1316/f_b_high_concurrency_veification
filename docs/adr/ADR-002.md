# ADR-002: 高併發庫存扣減場景的分片驗證

**狀態：** 已接受 (Accepted)  
**關係：** 解決 ADR-001 在橫向擴展時的熱點 Key 瓶頸

## 1. 背景 (Context)
雖然 ADR-001 透過非同步佇列與快取防禦解決了併發阻擊問題，但在 Redis Cluster 環境中，單一活動商品的庫存 Key 會導致 **資料傾斜 (Data Skew)**。當壓測或搶票流量進一步擴張，單一 Redis 節點的 CPU 與網路頻寬將成為系統天花板，無法單靠增加節點來線性提升系統的整體效能。

## 2. 決策 (Decision)
捨棄「單一總庫存 Key」的集中式設計，改採 **「Master-Worker 分層補給架構」**：

*   **Master Inventory (總倉)**：
    存放於獨立 Redis Slot 節點，僅負責接收與處理各 Worker 節點的配額補給請求，不直接面對終端使用者的搶票扣減。
*   **Worker Shards (分片倉)**：
    將活動總庫存拆分為多個分片，動態分發並快取於多個 Redis 叢集節點。後台 Worker 在處理 `ticket:request:queue` 中的請求時，依據路由原則（如使用者 ID 雜湊）向對應的 Worker Shard 執行 Lua 扣減。且 Lua 中的所有 Key 必須在同一個 Slot 內。
*   **動態補給邏輯**：
    當特定 Worker Shard 的快取庫存水位低於預設閾值（Threshold）時，自動觸發非同步補給程序向 Master 申請新的庫存配額，回填至該分片。

## 3. 理由 (Rationale)
1. **橫向擴展性 (Horizontal Scalability)**：扣庫存請求被分散至不同的 Redis 實體節點處理，真正實現透過增加硬體節點來線性提升系統總容量與 TPS。
2. **熱點消除 (Hotspot Elimination)**：避免了數萬筆併發連線同時競爭同一個 Redis Key 造成的鎖定開銷與 CPU 瓶頸。
3. **系統韌性 (Resilience)**：即使單個 Worker Shard 節點發生故障，僅影響分配至該分片的部分庫存，系統其餘節點仍能維持正常售票運作。

## 4. 後果 (Consequences)
*   **優點**：
    *   具備高度的橫向擴展能力，能承受極大規模的分散式流量衝擊。
*   **挑戰與配套**：
    *   **庫存碎片與分配問題**：在活動售罄末期，可能會出現 Master 總倉已空，但各個 Worker Shard 仍有零星餘票的分配不均問題（庫存碎片）。必須實作「碎片回收與重分配機制」，在活動末期將餘票收回並重新集中分配。
