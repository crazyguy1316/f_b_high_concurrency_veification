# Test_Plan.md: 系統壓力測試與驗證計畫

## 1. 測試目標 (Test Objectives)
本計畫旨在利用 **k6** 於實體 **Docker** 容器化環境下，驗證搶票系統在高併發極限負載時的防禦、排隊、扣庫存與資料對帳能力：
*   **邊緣限流與防刷驗證**：驗證最前端 Nginx 反向代理網關的動靜分離、惡意流量清洗與防刷阻斷率。
*   **身份驗證與非法攔截**：驗證 App Server 的 Token 校驗模組，確認非法/登入失效 Token 被 100% 拒絕。
*   **資料一致性與防超賣**：確保不論高併發負載多大，實體 MySQL 成交量與快取扣減量均精確吻合，超賣率為 0.00%。

---

## 2. 壓力測試情境 (Load Testing Scenarios)

### S1: 萬人極速搶票與混合測試 (The Big Bang & Mixed Request Test)
*   **描述**：模擬開賣瞬間，大量使用者同時湧入，且請求中夾雜無效 Token 與異常存取。
*   **壓測工具**：**k6**
*   **負載參數**：
    *   虛擬使用者 (VUs)：10,000 VUs。
    *   加壓模式：1 秒內完成 Ramp-up。
    *   **流量混合比例**：
        *   **80% 合法搶票請求**：攜帶正確 `member_id` (int) 與有效 Token。
        *   **10% Token 驗證失敗請求**：攜帶無效/過期 Token（預期回傳 `401 Unauthorized`）。
        *   **10% 惡意刷票 / 超頻請求**：單一客戶端 IP 頻率大於 2r/s（預期由網關回傳 `403 Forbidden` / `503 Service Unavailable`）。
*   **預期結果**：
    *   網關層正確執行動靜分離，靜態 HTML/JS 不消耗 App 伺服器資源。
    *   所有 401、403、503 請求被邊緣精確截流，未進入消息佇列。
    *   MySQL 資料庫最終成交筆數精確等於 Redis 初始庫存（例如 20,000），**無超賣，無丟單**。

### S2: 邊緣限流閥值測試 (Rate Limiting Threshold)
*   **描述**：專門驗證 Nginx 反向代理網關對 `/api/v1/tickets/*` 的令牌桶/漏桶演算法。
*   **測試方法**：
    *   使用單一來源 IP 以 50 VUs 併發速率持續發送 API 請求。
*   **預期結果**：
    *   第 1 與第 2 筆合規請求成功放行。
    *   後續超出閥值（例如 2r/s 限制與 Burst 限制）的請求，由 Nginx 直接於邊界予以阻斷，回傳 `403` 或 `503`，保護後端不致崩潰。

### S3: Worker 崩潰與斷線落庫對帳測試 (Fault & Reconciliation)
*   **描述**：驗證 MySQL 落庫失敗時的系統韌性與最終一致性。
*   **測試步驟**：
    1.  於 k6 壓測中途，人工手動關閉或重載 MySQL 資料庫容器，造成 Worker 寫入實體訂單失敗。
    2.  待資料庫恢復後，執行對帳腳本。
*   **預期結果**：
    *   系統能透過監控發現 MySQL 訂單數 < Redis `ticket:success:orders` 暫存數。
    *   透過對帳 CLI 抓出未落庫的 `member_id`，自動或手動補單完成，MySQL 訂單總數回復至最終一致性。

### S4: 前端網頁 E2E 瀏覽器驗證測試 (Frontend Web E2E Browser Testing via k6-browser)
*   **描述**：考量單機硬體環境限制（無法同時啟動過多實體 Chromium 進程），安排極低併發的真實瀏覽器行為 E2E 測試。透過 k6-browser 模組啟動瀏覽器載入網關託管的前端搶票網頁，以使用者視角模擬按鈕點擊，驗證完整 UI 互動、轉跳邏輯與錯誤訊息渲染。
*   **測試工具**：**k6 (k6-browser / k6/x/browser)**
*   **測試參數與情境驗證**：
    *   併發使用者：1 ~ 5 VUs (低負載，著重功能性與渲染斷言)。
    *   **案例 1：購票成功流程** ── 攜帶有效 Token 點擊搶票按鈕，驗證頁面是否收到 202 Queue 狀態，並正確顯示「排隊中」及最終「搶票成功」提示。
    *   **案例 2：憑證無效 (401) 流程** ── 在前端寫入無效 Token 進行搶票，斷言前端網頁能正確補獲 401 響應並跳轉或提示「登入逾期/請重新登入」。
    *   **案例 3：頻率超限 (403/503) 流程** ── 在單一瀏覽器快速、高頻連點按鈕，觸發網關限流，斷言前端正確渲染「操作過於頻繁」之錯誤提示。
    *   **案例 4：售罄 (Sold Out) 流程** ── 於系統售罄後載入網頁，斷言搶票按鈕變更為 Disabled 灰色狀態，且文字顯示為「已售罄」。

## 3. 測試環境與配置 (Environment & Configurations)

*   **容器化部署拓撲 (Docker Compose)**：
    *   `nginx`：做為最前端反向代理網關。
    *   `app_server`：執行 FastAPI (處理 API 與 Token 驗證)。
    *   `redis`：做為消息佇列與 Lua 快取扣減。
    *   `db_mysql`：實體訂單落庫。
    *   `background_worker`：異步消費佇列，落庫 MySQL。
*   **測試主機**：單機 Docker 環境，需在壓測期間持續觀測 CPU、I/O 及記憶體健康度。

## 4. 關鍵效能指標 (Acceptance Criteria / KPIs)

| 評量指標 | 通過門檻 (KPI Criteria) | 說明 |
| :--- | :--- | :--- |
| **超賣率 (Overselling Rate)** | **絕對 0.00%** | MySQL 成功訂單數 <= 初始釋出票數，無溢出。 |
| **Token 失敗攔截率** | **100%** | 所有夾雜在壓測中的無效 Token 必須全數攔截回傳 401。 |
| **網關防刷阻斷率** | **100%** | 超頻之惡意連線必須被 Nginx 阻斷回傳 403 / 503。 |
| **資料對帳一致性** | **100%** | (Redis 成功名單雜湊表數量 == MySQL 成功訂單筆數)。 |
| **API 響應時間 (P95)** | **< 500ms** | 排除被網關直接阻斷限流的無效連線後之響應。 |

## 5. 測試產出與報告規範 (Reporting)

測試執行完成後，需產出 k6 效能報告，並包含以下對帳審計斷言：
1.  **k6 HTTP 請求狀態分佈**：確認 `401`（驗證失敗）、`403/503`（網關限流）與 `202/200`（成功入隊/落庫）之佔比符合壓測比例。
2.  **資料庫對帳紀錄**：提供對帳腳本輸出的日誌，證明 MySQL 成交量與測試賣出量一致。
3.  **效能曲線圖**：包含 VUs、TPS、Latency 與 Error Rate 走勢。
